# 61-iac-delivery-incident-driftwatch

Portfolio-grade, deterministic demo that makes infrastructure changes **reviewable**, deployments **safer**, and incidents **boring** by generating a single audit-friendly report from small, realistic inputs.

This repo targets three failure modes that show up in real orgs:

1) **Infrastructure drift & fragile automation**: “it worked in staging” because environments silently diverged or manual changes bypassed IaC.
2) **Delivery friction**: slow/flaky pipelines and risky releases because guardrails are missing or inconsistent.
3) **Reliability under on-call pressure**: poor signal-to-noise, unclear ownership, and missing runbooks increase MTTR.

It also maps cleanly to AI-service expectations (from the Zillow JD): scalable/secure service + data infrastructure, latency/scalability pressure, and operational efficiency via testing + observability.

## Architecture (inputs → checks → outputs → runbooks)

- Inputs: `examples/` (TOML + JSONL)
  - `examples/baseline/`: compliant inputs (used by `make lint` / CI)
  - `examples/drifted/`: intentionally drifted inputs (used by `make demo`)
- Checks: `python3 -m portfolio_proof validate --examples examples/baseline`
- Output: `artifacts/report.md` (gitignored) generated by `python3 -m portfolio_proof report --examples examples/drifted`
- Runbooks: `docs/runbooks/` linked from the report

## Quick start

```bash
make setup
make demo
```

Open `artifacts/report.md` to see:

- Drift summary (desired vs observed), risk scoring, and remediation steps
- CI/CD guardrail validation (required gates, approvals, rollback readiness)
- Reliability checks (SLO coverage including **latency SLO**, incident MTTR, runbook readiness)

## Demo (what to look for)

The generated report is intentionally structured around the pain points:

- Drift: missing tags, unmanaged resources, and config deltas that would break reproducibility.
- Delivery: missing “plan before apply”, missing approvals, and missing rollback hooks.
- Incidents: whether SLOs exist, whether runbooks exist, and whether MTTR is improving.

## Security (and what’s out-of-scope)

- No secrets are required. `examples/` are sanitized.
- `validate` includes basic secret-pattern scanning to fail fast if sensitive strings leak in.
- Out-of-scope: real cloud API calls. The demo uses realistic inputs and produces guardrail decisions deterministically (you can wire it to AWS/GCP/Azure later without changing the core rules).
